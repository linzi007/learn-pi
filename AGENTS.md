# Learn Pi 项目协作规范

## 项目定位

本项目是面向中文读者的非官方 Pi 原理学习仓库，目标是通过短小、可运行、可验证的文档与 Demo，解释 [Pi Agent Harness](https://github.com/earendil-works/pi) 的核心设计和执行链路。

- 本项目不是 Pi 的 fork、替代实现或官方文档镜像。
- 可以参考 `learn-claude-code` 的分课方式，但不得复制其课程代码或累积快照结构。
- 讲解必须以 Pi 的真实源码为依据；教学简化必须明确标注，不能写成上游的真实行为。
- 当前第一版只维护简体中文，不同步维护英文、日文等翻译版本。

## 教学基线

- 第一版默认以 Pi `v0.80.6` 为教学基线，对应提交 `2b3fda9921b5590f285165287bd442a25817f17b`。
- 源码链接必须优先使用固定 tag 或 commit 的 GitHub permalink，不使用会随 `main` 漂移的链接。
- 若必须介绍基线之后的能力，应明确标记为“基线后变化”，并记录核查时使用的提交。
- 本地 Pi 和 `learn-claude-code` clone 只用于查阅。公开仓库的安装、运行和测试不得依赖本机兄弟目录或绝对路径。

## 语言与技术栈

- Demo 默认使用 TypeScript、ESM 和 Node.js 22，与 Pi 上游保持一致。
- 文档、代码注释和用户可见输出默认使用简体中文。
- 代码标识符、API 名、命令、协议字段和原始错误信息保留英文，不做生硬翻译。
- 依赖应保持精简并使用精确版本；提交 lockfile，保证课程可复现。
- 默认 Demo 必须离线可运行，不要求 API Key、付费模型或个人 Pi 配置。
- 需要模型响应时优先使用 Pi 提供的 faux provider 或项目内确定性 fake；真实 provider 只能作为显式可选路径，并通过环境变量配置。

## 课程代码架构

课程必须从结构上避免“后面的课复制前面所有代码”的问题。推荐职责如下：

```text
src/
  core/                  # 稳定、最小的教学内核
  features/              # 按课程归属拆分的增量能力
  testing/               # fake provider、事件记录器和测试辅助
lessons/
  s01-topic/
    README.md
    demo.ts              # 薄入口，只装配并演示本课能力
    demo.test.ts
  s02-topic/
    README.md
    demo.ts
    demo.test.ts
```

必须遵守以下规则：

1. 每课只引入一个主要概念，课程入口只保留本课的装配和演示代码。
2. 禁止把前面课程的类、函数、类型、工具定义或 Agent Loop 复制到后续课程。
3. 跨课程复用的实现放入 `src/core/`、`src/features/` 或 `src/testing/`，后续课程通过 `import` 复用。
4. 禁止创建持续膨胀的 `common.ts`。共享代码必须按 `agent-loop`、`messages`、`tools`、`events`、`state`、`session` 等职责拆分。
5. `src/core/` 只能包含稳定基础机制，不得为了方便提前塞入后续课程才讲解的行为。
6. 增量能力应拥有明确的课程归属，例如 `src/features/s03-tool-execution.ts`。读者必须能直接定位“本课新增实现”。
7. 课程不得导入其他课程的 `demo.ts`。需要表达阶段性组合时，使用独立的 feature 或 stage builder，并保持依赖单向、无环。
8. 最终综合课也必须通过模块组合完成，禁止生成一个包含全部历史实现的全集文件。
9. 单课新增的核心教学代码目标为 50 至 150 行；单个课程入口目标不超过 200 行。超过 250 行必须拆分，并在评审中说明原因。
10. 修改共享模块后必须回归全部已发布课程。共享层的演进不能悄悄改变早期课程展示的语义。
11. 不为假设中的未来复用提前抽象。只有出现第二个真实调用者时，才把课程代码提取到共享模块。
12. 第一课和新概念的首次出现应优先保证单文件顺序可读；后续课程再通过共享模块复用，不能让读者为了理解主链路跳转多个 helper。

## 每课文档契约

每个 `lessons/sNN-topic/README.md` 的主线结构统一参考 `learn-claude-code`：

1. 标题、课程导航和一句话核心结论
2. `## 问题`：从无需术语即可理解的具体场景切入
3. `## 解决方案`：先用主图建立完整心智模型，再压缩成一句规则或一张小表
4. `## 工作原理`：严格按 Demo 的真实执行顺序拆解，每段代码都能直接定位到实际文件
5. `## 试一下`：给出运行命令、真实预期输出、可尝试输入和明确观察问题
6. `## 接下来`：用当前方案的一个具体缺口自然引出下一课
7. `<details><summary>深入 Pi 源码</summary>`：放固定 commit 源码定位、生产实现差异、错误路径和扩展阅读

主线先让读者获得直觉和成功体验，再解释术语和工程边界。学习目标、文件清单、测试契约和源码链接不得堆在正文开头。

README 中展示的最小代码必须自洽，并直接来自课程 Demo 的同一执行链。禁止文档展示一套伪代码、实际 Demo 再通过多个 helper 隐藏关键逻辑。

## 图片规范

- 每课至少提供一张位于课程 `images/` 目录的中文主图，紧跟在“解决方案”标题后。
- 主图必须包含具体输入、关键状态、判断或数据流和最终结果，只看图也应能复述本课核心机制。
- 复杂的流式、循环、并发、会话或源码调用关系应增加独立图示，不用一张图塞入所有信息。
- 图片优先使用仓库内静态 SVG，必须有清晰的 Markdown alt 文本和图下注释。
- 图中颜色用于区分输入、处理、判断、成功和错误，不做纯装饰。
- 图中的节点、术语和顺序必须与正文及 Demo 一致；代码变化后同步检查图片。
- 第一版只维护中文图片，不创建空的多语言副本。

## Demo 与验证

- 每课必须可以从仓库根目录独立运行，不能依赖先执行前一课产生的文件或状态。
- 可运行 Demo 只使用 Pi 的公开 package exports，不从已安装依赖中深度导入内部 `src/` 路径。
- 测试不得读取用户的 `~/.pi/agent`、真实认证信息、历史会话或全局设置。
- 会话、认证、模型注册和设置优先使用显式的 in-memory 实现。
- 每课至少有一个确定性的 smoke test 或断言，覆盖本课新增行为。
- 每个关键概念至少包含一个失败路径、边界条件或负向测试。
- 修改课程代码后运行该课程测试；修改共享代码后运行全课程验证。
- 文档中的命令和预期输出必须实际核验，不能只做静态推测。
- “读过源码”不算课程完成；必须存在可运行 Demo、测试结果或其他可复验证据。

项目脚本建立后，统一提供以下入口，避免每课自创命令：

```text
npm run lesson -- s01
npm run test:lesson -- s01
npm run verify
```

其中 `npm run verify` 应覆盖类型检查、格式检查、全部课程测试、课程依赖方向和明显重复代码检查。

## 源码研究规则

- 写原理说明或 Demo 前，先完整阅读相关 Pi 源文件及其类型、测试和公开导出，不根据文件名或 API 名猜测行为。
- 优先沿真实调用链研究，例如 `pi-ai -> pi-agent-core -> pi-coding-agent`，不要只摘取孤立函数。
- 明确区分公开稳定 API、内部实现、兼容层和实验性模块。
- 第一版主线聚焦 `pi-ai`、`pi-agent-core`、`pi-coding-agent` 和必要的 `pi-tui`；实验性的 orchestrator 默认放在附录。
- 如果上游行为与当前文档冲突，以锁定基线的源码和测试为准，并修正文档或 Demo。

## 开源边界

- 禁止提交 API Key、token、cookie、账号信息、真实会话、私人日志、内部域名和个人提示词数据。
- 禁止在公开内容中写入本机绝对路径。示例使用仓库相对路径或明确的占位符。
- `.env`、缓存、构建产物、编辑器配置和临时验证文件必须被忽略；只提交可公开的 `.env.example`。
- 不直接复用 Pi Logo 或其他品牌素材，除非已确认其单独授权。
- 引用少量上游代码时必须保留来源、许可证和版权信息；优先自行编写最小教学实现并链接原源码。
- 根 README 必须声明本项目非官方，并链接 Pi 上游项目。
- 首次公开发布前必须补齐 `LICENSE`、上游来源与版本说明、贡献指南和安全说明。

## Codex 工作方式

- 修改前先读取相关文件和上游实现，保持改动范围聚焦。
- 不修改本地的 Pi 或 `learn-claude-code` clone，除非用户明确要求。
- 不把个人学习进度、career 同步路径或本机环境写入公开仓库。
- 不为了减少文件数而隐藏课程增量；抽取公共代码后，README 必须给出清晰入口。
- 发现文档与 Demo、Demo 与测试、教学实现与上游源码不一致时，先修复一致性再继续新增课程。
- 未经用户明确要求，不创建 commit、不发布 package、不推送远端。
